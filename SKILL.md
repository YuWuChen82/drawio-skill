# drawio-skill

根据自然语言生成标准 `.drawio` XML 文件（兼容 draw.io / diagrams.net）。布局结构化、网格对齐（10px）。

## 何时激活

以下场景自动激活本 skill：

- 用户说"画图"、"帮我画"、"生成图"
- 提及 "draw.io"、"diagrams.net" 或要求导入 draw.io
- 请求"ER图"、"用例图"、"时序图"、"流程图"、"类图"
- 描述业务过程、数据库结构、系统交互或用户需求
- 提供表结构、字段、工作流或角色-用例描述

## 支持的图类型

| 类型 | 布局方向 | 核心元素 |
|------|---------|---------|
| **ER 图** | 网格（左→右 或 上→下） | 表 + 字段行 + 关系连线 |
| **用例图** | 左（Actor）→ 右（用例，按语义分层排列） | 火柴人 Actor + UseCase 椭圆 + include/extend/泛化连线 |
| **时序图** | 垂直，上→下 | 对象框 + 生命线 + 消息箭头 |
| **流程图/BPMN** | 左→右（或上→下） | 圆角矩形（开始/结束）+ 矩形（活动）+ 菱形（判断）+ 箭头 |
| **类图** | 网格布局 | 类框（名/属性/方法）+ 关系连线 |

## 执行步骤

### 步骤 1 — 确认类型和范围

若用户需求不明确，主动询问：

```
1. 图的类型：[ER 图 / 用例图 / 时序图 / 流程图 / 类图 / 其他]
2. 主要对象：[角色 / 系统 / 模块 / 数据表]
3. 核心内容：
   - ER：表列表 + 字段 + 关系 + 是否多模块
   - 用例图：角色列表 + 用例列表 + 模块划分 + include（必选）/ extend（可选）/ 泛化
   - 时序图：参与者 + 各参与者的操作 + 消息/返回
   - 流程图：开始 → 步骤 → 判断 → 结束
4. 输出文件名：[默认：{前缀}_{图类型}_NNN.drawio，按顺序编号，不覆盖旧文件]
```

### 步骤 2 — 布局原则：语义分层 + 分组（核心）

> **布局不仅是公式，更是语义。元素的位置由其含义决定，而非序号。**

#### 通用视觉层级

```
第一优先级（主视觉区）：核心业务流程、主实体、主参与者
    ↓
第二优先级（次要区）：管理/配置功能、次要参与者
    ↓
第三优先级（边缘区）：异常处理、边缘用例、辅助功能
```

- 核心元素居中或居左偏上；边缘元素靠边
- 同级元素均匀分布，不同级元素用留白区分
- **禁止**：所有元素平铺在一条水平线上；包含/扩展关系两端相距甚远

#### ER 图 — L1~L4 语义分层布局

**UML参考：** 实体按依赖层次从左上→右下排列，主键关系路径清晰。

```
布局结构：

  L1 基础实体 ───────────────────────────▶ L2 业务实体
  （用户/账户/角色）                         （商品/订单/帖子）
  左上角 x=40                                居中

  L3 附属/明细                              L4 配置/字典
  （订单项/评价/收货地址）                    （分类/标签/优惠券/权限）
  右部或下行                                 右下边缘

公式：
  行间距 = max(所有表高) + 80
  列宽   = max(所有表宽) + 80
  L层确定行号：L1→行0，L2→行1，L3→行2，L4→行3
  表x = 40 + 列号 × 列宽
  表y = 40 + L层行号 × 行间距
```

- **禁止**：L1表放右下；L4表放左上；跨层跳跃连线（L4直连L1）；表间距 < 60px

#### 多模块 ER 图 — 模块分组 + 跨模块 include/extend 关系

> **触发条件**：ER 图包含 2 个及以上独立业务模块（如"用户模块"+"订单模块"），或用户明确说"多模块"。
> **画布**：强制使用 `3200 × 2400`，模块之间预留 300px 留白。

**模块分组规则（与 L1~L4 布局互补）：**
```
  ┌─────────────────┐          ┌─────────────────┐
  │  模块A：用户模块  │          │  模块B：订单模块  │
  │  L1表（用户）    │          │  L2表（订单）    │
  │  L3表（地址）    │          │  L3表（订单项）   │
  └─────────────────┘          └─────────────────┘
          │ include（必选）               │ extend（可选）
          └──────────▶ 跨模块关系线 ──────┘
  （模块A → 模块B：include，虚线箭头）
```

**Include 关系（必选）：**
- **何时必须有**：当两个模块之间存在数据依赖时（如订单引用用户的 user_id），必须用虚线箭头表示模块间的引用依赖关系
- **样式（必须按此格式，不可拆分为独立节点）**：
  - 连线本身：`edge="1"`、`dashed=1`、`dashPattern=8 4`、`endArrow=open`、`endFill=0`、`strokeColor=#555`、`edgeStyle=entityRelationEdgeStyle`
  - `<include>` 标签必须写在连线的 `value` 属性中（`value="&lt;include&gt;"`），样式加 `edgeLabel;align=center;verticalAlign=top;fontSize=10;fontColor=#888;fontStyle=2`
  - **禁止**：将 `<include>` 写成独立的 `vertex="1"` 文本节点（draw.io 不会将其识别为连线标签）
- **连线规则**：从源模块的边界框右侧/下边缘出发，水平/垂直走到目标模块边界

**Extend 关系（可选）：**
- **何时可以有**：当某模块的表是另一个模块表的特例时（如"优惠订单"是"订单"的特例），使用 Extend
- **样式（必须按此格式，不可拆分为独立节点）**：
  - 连线本身：`edge="1"`、`dashed=1`、`dashPattern=8 4`、`endArrow=classic`、`endFill=0`、`strokeColor=#555`、`edgeStyle=entityRelationEdgeStyle`
  - `<extend>` 标签必须写在连线的 `value` 属性中（`value="&lt;extend&gt;"`），样式加 `edgeLabel;align=center;verticalAlign=top;fontSize=10;fontColor=#888;fontStyle=2`
  - **禁止**：将 `<extend>` 写成独立的 `vertex="1"` 文本节点
- **连线规则**：垂直或水平从子类指向父类，箭头指向基类

**模块间留白（防重叠核心）：**
```
模块A边界 ──────────────────┐
                             │  ≥300px 留白
          ┌─────────────────┘
          │
模块B边界 ──────────────────┘
（两模块边界之间至少留 300px，防止跨模块连线与模块内连线重叠）
```

**禁止**：跨模块连线穿过其他模块的主体区域；模块之间的连线与模块内连线使用同一通道；Include/Extend 箭头指向空白区域

#### 用例图 — 横向树形分组布局

> **⚠️ 强制：所有 Actor 必须为 `shape=umlActor`（火柴人），禁止使用矩形（rect）、圆形（ellipse/circle）、人形图标（person）等任何其他形状。**

**布局核心思想：先分类，再从左到右扇形展开，防止重叠。**

```
  ┌──────────────────────────────────────────────────────────────────────┐
  │ 步骤1：按语义将所有用例分为若干大类（如"商品管理"、"订单管理"、"用户管理"）│
  │ 步骤2：每个大类作为一个分支，从左到右水平排列                              │
  │ 步骤3：大类内的多个用例，垂直方向均匀分布（扇形展开）                        │
  └──────────────────────────────────────────────────────────────────────┘

布局结果示例：

  Actor1 ──┐
            ├── uc1_1 ── uc1_include ── uc1_extend
  Actor2 ──┤                                           （同组：垂直均匀分布）
            ├── uc2_1 ── uc2_include                   （组间距≥120px）
  Actor3 ──┤                                               （跨组连接走外侧）
            ├── uc3_1 ── uc3_extend
            │
  ActorSys ─┴── uc_sys_1
  （Actor均在左侧，火柴人对齐各自用例行；外部系统Actor在最下方）
```

**坐标计算（全部公式化）：**

1. **Actor**：x=40，y间距=80px（每行一个），宽30×高60
2. **用例分组**：
   - 组间距（水平）= 120px
   - 组起始x = 120 + 20 = 140
   - 组内UC垂直间距 = 60px（每UC一行，不重叠）
3. **单个分组内的UC布局**：
   - 第1个UC：x=140，y = 该行Actor的y值
   - 第2个UC：x=140，y+60
   - 第3个UC：x=140，y+120
   - 以此类推，垂直向下展开
4. **Include/Extend**：紧邻基UC右侧或下方（≤30px），不占用额外垂直空间
5. **跨Actor连接**：连接线走在分组外侧（不走分组内部）

**分组命名规则：**
- 分组本身不画框（无 System Boundary），仅通过坐标体现分组关系
- 同一 Actor 的所有用例归为一组，垂直排列
- 不同 Actor 的用例绝不在同一水平行上

**Actor XML（必须用 umlActor）：**
```xml
<mxCell value="" style="shape=umlActor;verticalLabelPosition=bottom;verticalAlign=top;html=1;outlineConnect=0;fillColor=#fff;strokeColor=#333;strokeWidth=2;" vertex="1" parent="1">
  <mxGeometry x="40" y="{80 + (n-1)*80}" width="30" height="60" as="geometry"/>
</mxCell>
<mxCell value="Actor名称" style="text;html=1;strokeColor=none;fillColor=none;align=center;verticalAlign=top;fontSize=11;fontColor=#333;" vertex="1" parent="1">
  <mxGeometry x="10" y="{145 + (n-1)*80}" width="90" height="20" as="geometry"/>
</mxCell>
```

**Include/Extend 布局规则（紧邻基UC，不产生额外垂直间距）：**
```
Include（水平）：基UC右侧 → includeUC右侧，y=基UC中心y
  基UC ──▶ includeUC ──▶ includeUC2

Extend（垂直下方）：基UC正下方，x=基UC中心x，y+80
        基UC
         │
         ▼ extend1
         ▼ extend2
（Extend不占水平空间，不影响同组其他UC的布局）
```

**用例椭圆 XML：**
```xml
<mxCell value="用例名称" style="shape=ellipse;html=1;whiteSpace=wrap;fillColor=#fff;strokeColor=#555;strokeWidth=1.5;align=center;verticalAlign=middle;fontSize=12;fontColor=#222;" vertex="1" parent="1">
  <mxGeometry x="{140 + (col)*80}" y="{80 + (row)*60}"
    width="{max(文本长度px + 60, 120)}" height="50" as="geometry"/>
</mxCell>
```

**禁止**：Actor不在左侧竖排；多个Actor的用例挤在同一水平行；Include/Extend占用额外垂直空间导致同组用例重叠

> **禁止**：把所有用例平铺在一个水平带上；Include/Extend 用例远离基用例（> 30px）；不同 Actor 的用例混杂无分组

#### 时序图 — 对象距离分层布局（BCE 参考模式）

**UML参考：** 从左到右代表"与发起者距离"递减——Actor → 边界/接口 → 控制/服务 → 实体/仓库 → 外部系统。

```
布局结构（从左到右）：

  Actor/User ──▶ 前端UI/接口 ──▶ 业务服务/控制器 ──▶ 数据库/Repo ──▶ 外部系统
  (发起者)       (Boundary)       (Control/Service)    (Entity)           (External)

对象间距：列宽 = max(该列文本最长px + 40, 100)
对象框 y=40，高50
消息y：y = 90 + 消息序号×60（序号1→90，序号2→150...）
生命线：从对象框底部(y=90)延伸至最末消息y+40
```

- 自调用：嵌套在对象生命线上（灰框包裹，y随消息序号递增）

> **禁止**：把数据库和前端随意排列（应有BCE顺序）；外部系统和内部服务混杂

#### 流程图 — 泳道分组 + 主干居中布局

**UML参考：** 同一泳道 = 同一角色/同一系统/同一职责；主流程居中，异常路径靠边。

```
垂直泳道布局（按职责）：

  ┌──────────────┬──────────────┬──────────────┐
  │  前端/发起者   │  后端/业务     │  数据/外部    │
  │  ──────────  │  ──────────  │  ──────────  │
  │  输入节点      │  处理节点      │  数据库节点    │
  │  开始椭圆      │  活动矩形      │  结束椭圆      │
  └──────────────┴──────────────┴──────────────┘
  泳道分隔线：strokeWidth=2，垂直分隔

主干流（水平左→右）：
  节点间距：水平≥60px，垂直≥80px
  判断分支：Yes→右(x+宽+60) 或 下(y+高+80)，No→反向
  合并：实心圆，直径30px

异常/错误分支：
  放在主干下方，黑色虚线框标注
```

- **禁止**：没有泳道所有活动混在一线；判断节点后缺少Yes/No两个分支；错误路径无标注

**UML参考：** 同包类在一起；继承树父上子下（垂直）；依赖箭头从客户端指向供应者。

布局结构：

  包A（domain层）：
    ┌─────────────────────┐
    │   User (父类)         │  ← 父类在顶部
    │   ─────────────────  │
    │   - id: Long         │
    │   + getId(): Long    │
    └─────────────────────┘
              │
    ┌──────────┴──────────┐
    │ Admin               │ │ Customer          ← 子类在父下方，垂直排列
    └─────────────────────┘ └─────────────────────┘
    （子类与父类列对齐，继承连线垂直向上）

  包B（infrastructure层）：
    ┌──────────────┐  ┌──────────────┐
    │ UserRepo     │  │ OrderRepo    │  ← 同包内横向排列，间距40px
    └──────────────┘  └──────────────┘

  包边界（可选虚线框）：
    strokeColor=#999，strokeDashPattern="3 3"
    宽 = max(包内所有类宽) + 60
    高 = 所有行高 + 80

- **禁止**：不同包的类混在一起；继承树横向排列；类间距随意

### 步骤 3 — 格式规范约束（必须遵守）

**宽高公式化原则（核心，禁止硬编码）：**
- 宽度 = max(内容文本长度 + padding, 最小宽度)
- 高度 = minHeight（可固定）或按行数/字段数计算
- 位置 = 起始坐标 + 序号 × (元素尺寸 + 间距)
- 具体公式见"步骤2"各图类型的布局结构说明

> **警告**：任何形状的 `width`、`height` 均不得写死为固定像素值，必须用公式计算。

**画布尺寸规范（按规模分级，有充足余量防止线条重叠）：**
| 规模 | pageWidth × pageHeight | 适用场景 |
|------|----------------------|---------|
| 小（< 5 个元素） | 1200 × 900 | 单角色 / 简单流程 |
| 中（5~15 个元素） | 1800 × 1200 | 普通 ER / 用例 / 时序 |
| 大（> 15 个元素） | 2400 × 1800 | 复杂 ER / 多角色时序 |
| 超大（> 30 个元素 / 多模块） | 3200 × 2400 | 多模块 ER / 多子系统用例 |

> **多模块场景强制使用超大画布**：当 ER 图包含多个独立模块（如订单模块、用户模块、商品模块分属不同包），或用例图包含多个子系统时，必须使用 `3200 × 2400` 画布，并为每个模块组预留至少 200px 的留白区域，防止跨模块线条与模块内线条重叠。

> **画布不够时的处理**：若图表生成后仍有线条重叠，先检查是否所有元素都遵守了"步骤2"的分层布局；若分层正确但画布仍不够，按上一档规模扩大画布。

**线条规整原则（分层正交路由，核心）：**

> 线条是图的骨架。路由不干净，布局再好也是乱的。

**一、四分层边线原则（所有图通用）：**

| 层次 | 含义 | 路由方向 | 示例 |
|------|------|---------|------|
| 第1层 | 主关系（核心依赖） | 水平直线或垂直直线 | L1→L2（ER）、Actor→主UC |
| 第2层 | 次要关系（扩展/引用） | 主线旁边平行，间距20px | Extend、1:1关系 |
| 第3层 | 辅助关系（Include/依赖） | 从主关系分支出来 | Include、虚线 |
| 第4层 | 自引用 | 大弧线绕开最近边 | 自引用 |

**二、锚点优先连接规则（线条从哪里进/出形状）：**

```
连接锚点优先级：
  水平布局（左右结构）→ 优先用右边缘（源）→ 左边缘（目标）
  垂直布局（上下结构）→ 优先用下边缘（源）→ 上边缘（目标）

禁止：从形状顶部/底部连向左右相邻元素（除非必须）
```

**三、总线布局（多条线共用路径）：**

当多个关系共享同一段路径时，使用总线布局：
```
总线（公共路径）: 一条水平/垂直干线，多条线在干线汇合后分开
  ┌──────────┐         ┌──────────┐
  │ 表A      │──────────│ 表B      │   ← 主线（水平）
  └──────────┘    │     └──────────┘
         │        │         ↑
         │    总线(垂直)    │
         ↓        ↓         │
  ┌──────────┐         ┌──────────┐
  │ 表C      │──────────│ 表D      │   ← 主线（水平）
  └──────────┘         └──────────┘
  （表A→表C、表B→表D 共享总线）
```

总线实现：在共享段插入拐点（via point），所有线在同一段走同一路径。

**四、分图线条层次规则：**

ER图：
- L1→L2 主关系走上方水平通道（第1层）
- L2→L3 走中间垂直+水平（第1层）
- L3→L4 走下方水平通道（第2层）
- 所有线不得穿过表主体

用例图：
- Actor→用例关联线：Actor右侧 → 用例左侧，水平直线，y对齐用例中心
- Include/Extend：基UC→扩展UC水平或垂直紧邻（间距≤30px），不得穿过其他用例
- 泛化树：垂直向上，子→父，父在子上方

时序图：
- 生命线：垂直居中（对象框正下方），不得偏移
- 消息箭头：水平，严格垂直对齐到消息序号位置
- 不得在对象之间画斜向消息线

流程图：
- 主干流：水平直线，拐点用直角（`rounded=1`）
- 分支：垂直向下或水平向右，直角路由
- 合并点：实心圆，不得从合并点画斜线

类图：
- 继承：垂直连线，父上子下
- 聚合/组合：水平或垂直，菱形在端点
- 依赖：虚线，客户端→供应者

**五、禁止的线条：**

- 斜线（所有图均禁止 `strokeColor` 无斜向路径）
- 折返线（线走到一半又折返）
- 穿越线（穿过无关形状的主体）
- 重叠线（两条线完全重合）
- 悬空线（箭头指向空白处）

**内容分布规范（禁止挤到一角）：**
- 核心元素居中或左上，边缘元素靠边
- 同行元素均匀分布；某行元素偏少（< 3个）时居中或分散排列
- 各级之间用留白区分

### 步骤 4 — 生成 XML

根据图类型，参照 `references/drawio_xml_format.md` 中的格式规范生成 XML。

### 步骤 5 — 输出

1. **确定输出路径 + 文件名**（全部带顺序编号，禁止覆盖旧文件）：
   - 文件必须写入**当前项目根目录**（而非用户文件夹）
   - 文件名格式：`{前缀}_NNN.drawio`
     - 前缀取自用户描述（去掉空格作为前缀），加上图类型
     - 例：用户说"画一个电商系统ER图" → `电商系统_ER图_001.drawio`
     - 例：用户说"生成用户登录用例图" → `用户登录_用例图_001.drawio`
   - 编号规则：从 `001` 开始，扫描同目录下已有匹配前缀的文件，自动找下一个可用编号
     - 例：已有 `电商系统_ER图_001.drawio` → 本次生成 `_002.drawio`
     - 例：已有 `_001`、`_003` → 本次生成 `_002`（填入空缺，不跳号）
     - 例：无任何匹配文件 → 本次生成 `_001.drawio`
   - 用户即使指定了文件名，也必须在末尾追加顺序编号（不允许覆盖任何已存在的文件）
   - 正确路径：`{当前工作目录}/{前缀}_{图类型}_NNN.drawio`
   - 错误路径：`C:\Users\{username}/{filename}.drawio`（用户文件夹，禁止）
2. **写入文件**：使用 `{当前工作目录}/{前缀}_{图类型}_NNN.drawio`
3. **回复内容**：
   - 图类型和元素概要
   - 核心结构说明
   - 完整文件路径（告诉用户在哪里找到文件）
   - 导入方式："draw.io → 文件 → 打开 → 选择 .drawio 文件"
   - 是否有需要调整的地方

## 输出标准

### 通用规则

- 文件路径：`{当前工作目录}/{前缀}_{图类型}_NNN.drawio`（项目根目录；全部带顺序编号，不覆盖旧文件）
- 外层包装：`<mxfile host="app.diagrams.net">`
- 画布：`page="0"`（无页面背景=纯透明）、`gridSize="10"`、`guides="1"`、`connect="1"`、`arrows="1"`
- 字体：`fontFamily=Helvetica`
- 所有坐标和尺寸均为 10 的倍数

### ER 图

- 表宽：`max(最长字段行字符数×8 + 20, 180)`（不得写死）
- 表高：`30 + 字段数 × 18`
- 关系连线：`edgeStyle=entityRelationEdgeStyle`
  - 1:N：实线 + `endArrow=ERmany`
  - 1:1：实线 + `endArrow=ERone`
  - 自引用：虚线 + `dashed=1`

### 用例图

- 角色：`shape=umlActor`（火柴人，禁止用矩形/圆形等替代）
- 用例椭圆：宽 `max(文本长度+60, 120)`，高50（不得写死）
- 关联：实线，无箭头（`endArrow=none`）
- Include/Extend：虚线，`endArrow=open`，标签 `<include>` / `<extend>`；Include UC与基UC间距≤30px
- 泛化：实线，`endArrow=classic`（实心三角）
- 角色↔用例间距：≥60px

### 时序图

- 列宽：`max(对象文本最长px + 40, 100)`（BCE顺序，不得写死200px）
- 对象框：y=40，宽按文本计算，高50
- 生命线终点：`y=90 + 消息数×60 + 40`（动态）
- 同步消息：实线 + 实心三角 `endArrow=block`，`endSize=10`
- 异步消息：实线 + 空心三角 `endArrow=open`
- 返回：虚线 + 空心三角

### 流程图

- 活动节点：宽 `max(文本长度+40, 120)`，高按行数计算（不得写死160×60）
- 判断菱形：最小120×80
- 开始/结束椭圆：宽按文本，高固定40
- 连接箭头：`endArrow=classic`
- 分支标签："是/Y" → 右；"否/N" → 下
- 合并：实心圆

## 禁止事项

- 不生成用户未要求的图类型
- 不跳过确认步骤
- 不使用非标准 XML 格式
- 不输出纯文本代替 XML 文件
- 不将大量内容塞入单个图（保持可读性）
- **禁止画布滥用**：尺寸远超内容所需；内容堆在画布一角；连线之间相互交叉
- **禁止硬编码宽高**：必须使用公式计算
- **禁止混乱布局**：元素平铺一线；Include/Extend 关系两端分离超过30px；无泳道无分层
- **禁止元素重叠**：Include/Extend 用例之间互相重叠；任何用例/节点覆盖其他用例或节点的主体；连线穿过形状主体
- **禁止彩色**：所有颜色值必须为灰度（#000/#333/#555/#999/#ccc/#fff）；禁止使用彩色（绿/黄/蓝/红/橙等）
- **禁止非火柴人 Actor**：参与者形状必须为 `shape=umlActor`（标准 UML 火柴人），禁止使用矩形/圆形/任何其他形状替代
